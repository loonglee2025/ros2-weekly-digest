# ROS2 Weekly Digest — 2026-06-08 至 2026-06-14

> **总判断**：本周 ROS2 社区延续 Lyrical Luth GA 发布后的平稳消化期，核心框架代码活跃度依然偏低，但社区工具创新与生态项目修复密集涌现。最值得提高跟踪权重的三个方向是：**被动式无侵入观测（ros2probe）**、**Gazebo 滚动发布模型（Rotary）**与 **Nav2 行为树配置正确性加固**。工业界对 ROS2 的采纳速度明显加速（Fanuc、Comau、Schunk），ROSCon Global 日程已公布，社区进入“功能可用→可观测、可运维、可规模部署”的转型阶段。

## 本周关键信号

1. **ros2probe 发布：用 eBPF 实现无 probe effect 的 ROS2 流量观测**（6 月 12 日）
   - 来自 DGIST 的 ros2probe 在内核层通过 eBPF 被动捕获 RTPS 报文，不加入 DDS domain、不创建 DataReader。在接近链路饱和场景下，rosbag2 可导致真实 subscriber 丢包率高达 **75.5%**，而 ros2probe 为 **0%**，且 CPU 和内存占用分别降低 **7 倍**和 **28 倍**。工程意义：这是 ROS2 观测工具的范式级突破，对高带宽实时场景（自动驾驶、高速视觉、高频控制）的数据记录与调试具有根本性价值。CLI 与 ros2 完全兼容（`ros2` → `rp`），录制格式为 MCAP。需继续跟踪其 SHM 共享内存回退策略、Zenoh 支持路线图，以及是否会被纳入官方工具链。

2. **Gazebo 正式发布 Rotary 滚动版**（6 月 11 日）
   - Gazebo 引入 Rotary 作为 nightly 滚动发布线，所有开发直接切到 `main` 分支，变更按需 backport 到稳定版。这与 ROS Rolling 的发布模型一致，目标是提升开发速度、简化长短期支持版本的发布流程。关键约束：从 Gazebo Jetty 起不再支持并行安装（Rotary 会覆盖 Jetty），对 CI 和开发环境管理带来直接影响。工程意义：仿真与机器人框架的发布节奏趋于一致，但插件 ABI 兼容性风险上升。需关注 Rotary 与 ROS2 Rolling 的集成进度（跟踪 issue #1475）。

3. **Nav2 本周 9 个 commits，主线转向“配置正确性加固”**
   - 核心变更包括：MPPI 控制器增加**轴延迟补偿与命令历史回放**（#6154），这是高频控制回路的精度修复；全局移除 mypy 类型注解（#6176），说明 Nav2 在简化类型维护负担；XML BT 节点验证机制（#6130）防止运行时加载无效行为树节点；AxisGoalChecker 修复（#6192）与 CI 升级。工程意义：Nav2 当前主线不是新增大功能，而是“修复控制器联动边界”和“加固行为树配置正确性”，这对生产环境的稳定性更关键。MPPI 的轴延迟补偿直接影响轨迹跟踪精度，建议使用 Nav2 做精确运动控制的团队关注。

4. **ros2_control 修复 async 模式纯虚函数崩溃与 Qt API 兼容性问题**
   - 修复了硬件组件在 async 模式下触发纯虚函数调用导致崩溃的问题（#3321），同时修复 Qt API 兼容性问题（#3367）并增加 spawner 测试超时（#3381）。工程意义：async 模式是 ros2_control 面向高频率控制回路的重点路径，该修复直接消除了生产场景下的段错误风险。建议使用 async 模式的用户确认当前发行版是否已包含此修复。

5. **rclnodejs 2.0.0 发布 typed Web SDK，支持 Lyrical**
   - 在 Lyrical GA 后紧跟发布，提供浏览器端 typed SDK + 能力声明运行时（`web.json`），未暴露的能力直接拒绝（`not_exposed`）。支持 HTTP fallback，AI agent 可通过 plain HTTP 调用 ROS2 能力。工程意义：这是“AI + ROS2”桥接生态的重要基础设施，其安全设计（能力白名单）值得借鉴。2.1.0-beta.0 已跟进发布，说明迭代节奏快。

6. **工业界 ROS2 采纳加速信号密集**
   - Fanuc 上线 ROS2 专属支持页面；Comau 和 Schunk 在 ICRA 2026 公开宣示 ROS2 支持；ROS-Industrial 年会与 Automate 展会同期举办。工程意义：ROS2 在工业机械臂和自动化场景的渗透率正在从“早期采纳”进入“主流供应商背书”阶段，这对生态长期健康是结构性利好。

7. **ROSCon Global 2026 日程公布，早鸟票截止 6 月 12 日**
   - 多伦多会议日程已上线，演讲阵容已确定。新增长区域 ROSCon：意大利（11 月 3 日，博洛尼亚）。工程意义：社区活动密度持续增加，区域 ROSCon 的扩展说明 ROS2 的地理覆盖正在从北美/欧洲向亚洲深化。

## 工程化观察

- **回归风险**：本周核心框架无大规模 sync，风险较低。但 ros2_control async 纯虚函数错误说明异步路径的测试覆盖仍有盲区，建议团队在使用 async 模式时确认修复状态。Nav2 的 XML BT 节点验证机制（#6130）可能会影响现有行为树的加载行为——如果团队使用了自定义 BT 节点但未按新规范注册，可能遇到运行时错误。
- **CI / 构建**：Nav2 升级了 `actions/checkout` 版本并修复 CI；MoveIt2 新增 Rolling-on-Resolute CI 作业；gz-sim 升级 pre-commit action。说明生态项目对 CI 基础设施的维护保持活跃。Discourse 上关于“ROS2 最烦人工作流”的讨论中，编译慢（rclcpp 头文件膨胀、colcon 扫描慢）和跨平台构建（ARM 交叉编译）仍是高频痛点。
- **可观测性**：ros2probe + ROSscope（上周）+ ros_babel_fish_tools（本周）同时出现，说明 ROS2 社区正从“功能可用”向“可观测、可调试、可运维”过渡。这是大规模部署和 Fleet 管理的必经阶段。ros2probe 的 eBPF 方案尤其值得在高性能场景下优先评估。
- **Windows 开发体验**：上周提到的 Ros2Simple 预编译 Humble SDK 仍在社区讨论中，本周无新进展。Windows 原生 ROS2 开发体验长期是痛点，但非官方方案的长期维护承诺仍需验证。
- **安全性**：gz-sim 的资源路径限制（#3590）缓解了一个潜在的安全问题；rclnodejs 的能力白名单机制（`not_exposed`）为 Web 暴露 ROS2 接口提供了安全边界。这两个方向说明安全设计正在从“事后修补”转向“前置声明”。

## 前沿议题

1. **被动式无侵入观测可能成为高性能场景标配**
   - ros2probe 的 eBPF 方案不仅适用于 rosbag2 替代，也可用于实时质量监控、异常注入检测和网络层调试。若其 Zenoh 支持得到验证，有望成为 ROS2 下一代标准调试工具。其 SHM 回退策略（短生命周期 shadow subscriber）是务实的工程折中，但失去了无侵入保证。

2. **AI + ROS2 的桥接生态进入产品化阶段**
   - AgenticROS（连接 OpenClaw/Claude/Gemini）和 rclnodejs 2.0.0（typed Web SDK + HTTP fallback）都在推进“AI 代理直接操作 ROS2”的范式。rclnodejs 的能力声明机制（未暴露的能力直接拒绝）是一个值得借鉴的安全设计。下一步应关注这些桥接工具在真实机器人上的延迟、可靠性和安全审计结果。

3. **Gazebo Rotary 与 ROS Rolling 的协同加深**
   - Rotary 的发布模型与 ROS Rolling 一致，仿真与机器人框架的发布节奏可能趋于同步。这对插件开发者意味着需要同时跟踪两个滚动版本的 API 变化。Jetty 起不再支持并行安装的约束，会对 CI 矩阵和开发环境管理带来直接影响。

4. **工业界采纳加速带来的生态质量压力**
   - Fanuc、Comau、Schunk 等主流工业自动化厂商公开支持 ROS2，说明 ROS2 正在从科研/教育场景进入工业主流。这会对稳定性、实时性、安全认证和长期支持提出更高要求，可能推动 ros2_control、MoveIt2 和 Nav2 在工业场景下的专项优化。
